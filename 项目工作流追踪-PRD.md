# Seegent 项目工作流追踪 — Feature Spec

## 1. 问题陈述

**当前痛点**：开发者在 AI 辅助编程中使用了多个智能体、CLI 工具、MCP 服务，但这些调用散落在操作日志和用量统计中，无法按「项目」和「工作流阶段」聚合查看。开发者不知道一个项目经历了哪些步骤、每个步骤用了什么工具、消耗了多少 Token。

**谁受影响**：任何在 Seegent 上做 AI 辅助开发的用户（当前只有你自己——苑畅）。

**不解决会怎样**：项目复盘靠回忆，Token 消费无法按阶段归因，无法回答"这个功能到底花了多少 AI 成本"。

## 2. 目标

| # | 目标 | 衡量标准 |
|---|------|---------|
| 1 | 以项目为维度聚合所有 AI 调用 | 打开项目页面能看到完整工作流 |
| 2 | 自动从日志导入 Token/Agent/工具数据 | 点「导入」后 90% 字段自动填充 |
| 3 | 工作流可视化（流程图） | 一眼看出哪些阶段完成、哪些进行中 |
| 4 | 每阶段 Token 和工具使用可量化 | 能回答"后端开发花了多少 Token" |
| 5 | 替代现有追踪面板 | 功能不退化，操作更少 |

## 3. 非目标（Non-Goals）

| 事项 | 原因 |
|------|------|
| 不监控 Seegent 之外的 AI 工具调用 | 技术上无法拿到外部进程的 token 数据 |
| 不做甘特图/日历视图 | v1 用流程图足够，后续迭代 |
| 不做多用户协作 | Seegent 是本地单用户应用 |
| 不实时自动记录所有调用 | 日志导入是手动触发的一步，避免数据噪音 |
| 不删除现有「用量」和「日志」模块 | 它们是底层数据源，保留作为数据参考 |

## 4. 用户故事

| 优先级 | 故事 |
|--------|------|
| P0 | 作为开发者，我希望创建一个项目并预设工作流模板（如「需求→设计→后端→前端→测试」），以便按阶段组织工作 |
| P0 | 作为开发者，我希望在每个阶段下添加步骤记录，注明用了什么 Agent/模型/工具、花了多少 Token，以便事后复盘 |
| P0 | 作为开发者，我希望一键从操作日志导入最近的 Agent 调用，自动生成步骤记录，以便减少手动填写 |
| P0 | 作为开发者，我希望看到项目的流程图，每个阶段标注状态和 Token 小计，以便快速把握进展 |
| P1 | 作为开发者，我希望修改工作流模板（增删改阶段），以便适配不同类型的项目 |
| P1 | 作为开发者，我希望在流程图中点击某个阶段展开查看详情，以便深入分析 |
| P1 | 作为开发者，我希望删除误导入的步骤或调整步骤所属阶段，以便修正错误 |
| P2 | 作为开发者，我希望按项目看到总 Token / 总费用 / 总耗时汇总，以便做成本分析 |

## 5. 数据结构

```json
{
  "projects": {
    "seegent": {
      "id": "seegent",
      "name": "Seegent",
      "description": "多人在线聊天智能体工作站",
      "createdAt": "2026-06-01",
      "workflow": {
        "template": ["需求分析", "方案设计", "后端开发", "前端开发", "测试联调"],
        "steps": [
          {
            "id": "step-uuid-1",
            "phase": "需求分析",
            "task": "新增追踪面板需求讨论",
            "date": "2026-06-27",
            "duration": 30,
            "agents": ["产品通"],
            "models": ["claude-sonnet-4"],
            "tools": { "cli": [], "mcp": [] },
            "tokens": 5000,
            "cost": 0.08,
            "notes": "确定需求方向，输出方案",
            "status": "done"
          }
        ]
      }
    }
  }
}
```

**字段说明**：

| 字段 | 类型 | 来源 |
|------|------|------|
| `id` | string | 项目唯一标识 |
| `name` | string | 手动输入 |
| `description` | string | 手动输入 |
| `workflow.template` | string[] | 创建项目时选择预设模板或自定义 |
| `steps[].phase` | string | 从 template 中选择，或日志导入时根据 agent 类型自动推断 |
| `steps[].agents` | string[] | 日志导入自动填，或手动输入 |
| `steps[].models` | string[] | 日志导入自动填 |
| `steps[].tools` | object | 日志导入自动填（toolsUsed 字段） |
| `steps[].tokens` | number | 日志导入自动填（totalTokens） |
| `steps[].duration` | number | 日志导入自动填（秒→分钟转换） |
| `steps[].cost` | number | 手动输入（日志不记录费用） |

## 6. 需求列表

### P0 — Must Have

| # | 需求 | 验收标准 |
|---|------|---------|
| R1 | 创建项目 | 用户输入项目名→选择工作流模板（预设 5-6 个或自定义输入，逗号分隔）→确认创建。预设模板包括：①软件开发的「需求→设计→后端→前端→测试」②内容创作的「选题→调研→写作→编辑→发布」③数据分析的「采集→清洗→建模→可视化→报告」④自定义（自由输入阶段） |
| R2 | 查看项目列表 | 展示所有项目卡片，每张显示项目名、阶段数、总 Token、最后活动时间 |
| R3 | 项目详情页（流程图） | 顶部显示流程图：每个阶段是一个节点，标注名称、状态图标（✅/🔄/⬜）、Token 小计。已完成的阶段用实线连接，未完成的虚线。点击节点展开该阶段的步骤列表 |
| R4 | 添加步骤 | 在某个阶段下添加步骤记录，表单字段同现有追踪面板（已优化默认值：日期默认今天、状态默认进行中、项目自动关联） |
| R5 | 从日志导入 | 点击「📥 从日志导入」→列出最近 20 条 Agent 调用→勾选条目→选择导入到哪个阶段（自动推断或手动指定）→一键生成步骤。自动填好的字段：日期、Agent、模型、工具、Token、耗时、任务描述 |
| R6 | 编辑/删除步骤 | 点击步骤卡片进入编辑，可修改所有字段。删除需确认 |
| R7 | 项目汇总统计 | 项目详情页顶部显示：总 Token、总费用、总耗时、完成阶段数/总阶段数 |

### P1 — Should Have

| # | 需求 | 验收标准 |
|---|------|---------|
| R8 | 修改工作流模板 | 在项目详情页编辑工作流模板：增删阶段、调整顺序。已存在的步骤自动跟随阶段名变更 |
| R9 | 删除项目 | 删除项目及其所有步骤数据，需二次确认 |
| R10 | 步骤拖拽调整 | 可以在不同阶段间拖拽步骤 |

### P2 — Future Considerations

| # | 需求 |
|---|------|
| R11 | 导出项目报告（Markdown/PDF） |
| R12 | 从项目模板库选择 |
| R13 | 流程图导出为图片 |

### 7.3 流程图节点设计

每个节点是一个圆角矩形卡片：
```
┌──────────────┐
│   需求分析    │  ← 阶段名
│      ✅       │  ← 状态图标（✅done/🔄doing/⬜todo）
│   5.0K        │  ← Token 数
│    30m        │  ← 耗时
│  (1 步骤)     │  ← 步骤数
└──────────────┘
```

- 已完成阶段：蓝色边框实线
- 进行中阶段：橙色虚线边框 + 脉冲动画
- 未开始阶段：灰色虚线边框
- 节点间用箭头连接
- 点击节点 → 下方展开该阶段的步骤列表

## 8. 技术实现

### 8.1 后端（seegent/server.py）

- 新增 `PROJECTS_FILE` 常量
- 新增 API：
  - `GET /api/projects` — 列出所有项目
  - `POST /api/projects` — 创建/更新项目
  - `DELETE /api/projects/:id` — 删除项目
  - `POST /api/projects/:id/import` — 从日志批量导入步骤
- 日志导入逻辑：读取 `/api/logs` → 按 workspace 推断项目 → 按 agent type 推断阶段 → 生成步骤

### 8.2 前端（seegent/static/index.html）

- 替换现有「追踪」模块为「项目」模块
- 新增组件：项目列表、流程图 SVG 渲染、阶段步骤列表、导入弹窗
- 复用现有：Modal 表单（字段 90% 不变）、TagInput 组件、CSS 变量

### 8.3 流程图渲染

使用内联 SVG 绘制流程图，无需外部库：
- 根据 `workflow.template` 数组动态生成节点
- 节点宽度根据文字长度自适应
- 自动换行（超过 5 个节点时换行显示）
- 节点可点击，触发阶段切换

### 8.4 日志导入推断规则

```
workspace 路径                 → 项目映射
/Users/.../Seegent      → seegent
/Users/.../LearnChinese   → learn-chinese

agent 类型                   → 阶段推断
产品通 / PM 类 agent        → 需求分析 / 方案设计
开发类 agent                → 后端开发 / 前端开发
测试相关 agent               → 测试联调
```

## 10. 与现有功能的共存策略

| 现有模块 | 处理方式 |
|---------|---------|
| 追踪（track） | **完全移除**，导航按钮和数据不再保留。旧数据若有价值可手动迁移 |
| 用量（usage） | 保留，作为底层数据参考 |
| 日志（log） | 保留，作为导入数据源 |
| 监测（data） | 保留，不冲突 |

## 11. 风险 & 注意事项

- 旧追踪面板数据（`.seegent-tracks.json`）不会被删除，只是前端不再展示。用户可手动在新项目中重建
- 日志导入的「阶段自动推断」可能不准确，提供手动修改入口
- 流程图在阶段数 > 6 时需要换行设计，避免节点堆叠
